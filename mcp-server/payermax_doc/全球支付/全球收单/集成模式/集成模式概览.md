# 集成模式概览

PayerMax收单服务提供了多种集成接入模式，满足商户的个性化结账、扣款等要求。在集成时您可以根据自身的能力和需求进行选择。

## 1. 收银台模式（Hosted Payment Page）

### 适用场景
- 快速集成，希望降低开发成本
- 不想自建收银台页面
- 无PCI-DSS认证资质
- 需要支持多种支付方式

### 核心特点
- **无需构造收银台页面**：PayerMax提供完整的支付页面
- **多端设备支持**：自动适配PC、移动端
- **商户无须PCI-DSS认证资质**：敏感信息由PayerMax托管
- **支持全量支付方式**：用户可在PayerMax页面选择所有可用支付方式

### 集成流程
1. 商户后端调用`/orderAndPay`接口创建订单
2. 获取`redirectUrl`跳转链接
3. 用户跳转到PayerMax收银台页面
4. 用户选择支付方式并完成支付
5. 支付完成后跳转回商户页面
6. 商户接收异步通知获取最终支付结果

## 2. 纯API模式（API Integration）

### 适用场景
- 希望在自己的收银台展示支付方式
- 需要控制支付方式的展示逻辑
- 支持钱包类支付直跳
- 有一定的前端开发能力

### 核心特点
- **支持钱包类直跳机构页面**：如支付宝、微信等可直接跳转
- **商户自行实现收银台页面**：完全自定义UI
- **支付方式是否展示可控**：可根据业务逻辑动态展示
- **无须PCI-DSS认证**：卡支付仍跳转PayerMax页面

### 集成流程
1. 商户前端展示自己的收银台和支付方式列表
2. 用户选择支付方式
3. 商户后端调用`/orderAndPay`接口，指定支付方式
4. 根据支付方式类型：
   - 钱包类：直接跳转到支付机构页面
   - 卡类：跳转到PayerMax卡支付页面
5. 支付完成后处理回调和通知

## 3. 前置组件模式（Drop-in Component）

### 适用场景
- 需要在自己页面内嵌入卡支付功能
- 希望自定义收银台UI
- 不想跳转到第三方页面
- 无PCI-DSS认证资质但需要卡支付

### 核心特点
- **商户须集成嵌入PayerMax Web组件**：在页面内嵌入支付组件
- **卡敏感信息PayerMax托管**：卡号、CVV等敏感信息不经过商户服务器
- **商户无须PCI-DSS认证资质**：符合PCI-DSS合规要求
- **更高的用户体验**：无需跳转，页面内完成支付

### 集成流程
1. 商户页面集成PayerMax Drop-in组件
2. 用户在组件内输入卡信息
3. 组件直接与PayerMax通信，获取token
4. 商户后端使用token调用支付接口
5. 完成支付流程

## 4. 链接支付模式（Pay By Link）

### 适用场景
- 无需开发，快速上线
- 通过社交媒体、邮件等渠道发送支付链接
- 临时性、一次性的支付需求
- 线下场景转线上支付

### 核心特点
- **可以不做代码开发**：通过管理后台或API创建链接
- **商户无须PCI-DSS认证资质**：使用PayerMax托管页面
- **灵活分享**：链接可通过任何渠道发送
- **快速部署**：无需技术集成即可使用

### 集成流程
1. 商户调用`/createPaybylink`接口创建支付链接
2. 获取支付链接URL
3. 通过邮件、短信、社交媒体等发送给用户
4. 用户点击链接进入PayerMax支付页面
5. 完成支付后接收通知

## 选择建议

### 快速集成（推荐新手）
- **首选**：收银台模式
- **备选**：链接支付模式

### 自定义体验
- **有开发能力**：纯API模式
- **需要卡支付且无PCI-DSS**：前置组件模式

### 特殊场景
- **社交电商、邮件营销**：链接支付模式
- **钱包类支付为主**：纯API模式
- **需要页面内完成卡支付**：前置组件模式

## PCI-DSS认证说明

PCI-DSS（Payment Card Industry Data Security Standard）是支付卡行业数据安全标准。

### 需要PCI-DSS认证的场景
- 商户服务器处理、存储或传输卡号、CVV等敏感信息
- 纯API模式中自建卡支付页面

### 无需PCI-DSS认证的模式
- ✅ 收银台模式
- ✅ 前置组件模式
- ✅ 链接支付模式
- ⚠️ 纯API模式（仅钱包类支付，卡支付跳转PayerMax）
